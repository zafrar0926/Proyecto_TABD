El Problema de Negocio

El Dolor

En F1, las decisiones se toman en milisegundos.

  • ❌ Datos dispersos y heterogéneos.
  • ❌ Formatos no optimizados para tiempo real.
  • ❌ Dificultad para cruzar históricos con telemetría en vivo.

La Solución

Una arquitectura híbrida que resuelve dos mundos opuestos:

  • ✅ OLAP (BigQuery): Análisis profundo de terabytes históricos.
  • ✅ OLTP (DynamoDB): Servicio de baja latencia para apps en vivo.

Objetivos SMART

Arquitectura de Alto Nivel

Diagrama de Componentes
MongoDB (Raw) BigQuery (Analytics) DynamoDB (Speed) Streamlit (UI)

Componente 1: Ingesta (MongoDB)

Landing Zone

Rol: Almacén Documental de datos crudos.

Colecciones: laps_enriched, weather, results.


¿Por qué MongoDB?

La telemetría cambia. Un esquema rígido SQL rompería el pipeline si la FIA agrega un nuevo sensor. El modelo JSON es natural para la ingesta.

// Ejemplo de Documento
{
"driver": "VER",
"lap": 45,
"telemetry": {
"speed": 310,
"throttle": 100,
"drs": true
},
"weather_impact": "dry"
}

Componente 2: Analítica (BigQuery)

Data Warehouse

Rol: Motor OLAP masivo.

Aquí ocurren las matemáticas pesadas:

  • Cálculo de tyreManagementIndex.
  • Funciones de ventana para consistencyScore.
  • Correlaciones estadísticas (SQL CORR()).

Ventaja Competitiva

El almacenamiento columnar permite escanear millones de vueltas en segundos. Sería ineficiente hacer estos cálculos agregados en Mongo o en tiempo real.

Componente 3: Servicio (DynamoDB)

Baja Latencia

Rol: Key-Value Store para la API.

Tabla: perfiles_pilotos

Clave: DriverID (ej. "VER")

Conclusiones y Futuro

Conclusión Principal

La arquitectura híbrida permite lo mejor de dos mundos: la flexibilidad de ingesta de MongoDB y la potencia analítica de BigQuery, sin sacrificar la velocidad de usuario final en DynamoDB.


Limitaciones

  • Ejecución Batch Manual.
  • Falta de Orquestación (Airflow).
